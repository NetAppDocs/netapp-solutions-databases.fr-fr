[role="lead"]
Cette solution fournit une vue d'ensemble et des détails sur le déploiement d'Oracle sur Google Cloud NetApp Volumes en tant que stockage principal de base de données via le protocole NFS, et Oracle est déployé en tant que base de données conteneur autonome avec dNFS activé.



== But

L'exécution de charges de travail Oracle exigeantes en termes de performances et sensibles à la latence dans le cloud peut s'avérer complexe. Google Cloud NetApp Volumes (GCNV) simplifie la migration et l'exécution de charges de travail Oracle exigeantes pour les professionnels des secteurs d'activité (LOB) et du stockage, sans modification de code. Google Cloud NetApp Volumes est largement utilisé comme service de stockage de fichiers partagé sous-jacent dans divers scénarios, tels que le nouveau déploiement ou la migration (lift and shift) de bases de données Oracle sur site vers Google Cloud.

Cette documentation présente la configuration des volumes GCNV DB dans la console Google Cloud et le déploiement simplifié de bases de données Oracle dans Google Cloud NetApp Volumes via des montages NFS à l'aide de l'automatisation Ansible. La base de données Oracle est déployée dans une configuration de base de données conteneur (CDB) et de bases de données enfichables (PDB) avec le protocole Oracle dNFS activé pour améliorer les performances. En outre, elle présente en détail une stratégie de sauvegarde, de restauration et de clonage de la base de données et fournit un kit d'automatisation pour la gestion des sauvegardes de bases de données Oracle sur Google Cloud.

Cette solution répond aux cas d’utilisation suivants :

* Déploiement automatisé d'une base de données Oracle sur Google Cloud NetApp Volumes dans Google Cloud avec le protocole NFS.
* Sauvegarde, restauration et clonage de la base de données Oracle sur Google Cloud NetApp Volumes dans Google Cloud.

=== Facteurs clés à prendre en compte lors du déploiement

* *Niveau de service et débit du pool de stockage GCNV.* GCNV propose quatre niveaux de service : Standard, Premium, Extreme et Flex. Pour les niveaux Standard, Premium et Extreme, le débit d'E/S est déterminé et fixé en fonction de la taille d'un volume de données. Le débit total d'E/S est limité par la taille du pool de stockage. Pour le niveau de service Flex, le débit d'E/S n'est pas fixé en fonction de la taille d'un volume de données, mais partagé entre tous les volumes de données et limité par la taille du pool de stockage. Ceci permet de gérer un volume de données de base de données plus petit avec des pics d'IOPS occasionnels. À titre indicatif, les niveaux de service Standard, Premium et Extreme offrent respectivement un débit de 16 Kio/s par Gio, 64 Kio/s par Gio et 128 Kio/s par Gio. Le niveau de service Flex, quant à lui, offre jusqu'à 5 Gio/s par pool de stockage avec des paramètres de performance personnalisés. Il est important de dimensionner correctement le niveau de service et le pool de stockage en fonction du débit d'E/S et des besoins en IOPS attendus pour la charge de travail de la base de données.
* *Architecture du stockage de la base de données.* Dans ce déploiement Oracle automatisé, nous provisionnons trois volumes de données pour chaque base de données afin d’héberger par défaut les fichiers binaires, les données et les journaux Oracle. Les volumes sont montés sur le serveur de base de données Oracle en tant que /u01 - binaire, /u02 - données, /u03 - journaux via NFS. Deux fichiers de contrôle sont configurés sur les points de montage /u02 et /u03 pour la redondance.
* *Configuration dNFS.* Grâce à Oracle dNFS (disponible depuis Oracle 11g), une base de données Oracle exécutée sur Google Compute Engine avec un stockage Google Cloud NetApp Volumes peut gérer nettement plus d’E/S que le client NFS natif. Le déploiement automatisé d’Oracle configure dNFS sur NFSv3 par défaut.
* *Sauvegarde par snapshot et Vault.* Plutôt que la sauvegarde traditionnelle de base de données RMAN, NetApp recommande de mettre en œuvre une sauvegarde par snapshot et vault, efficace en termes de stockage et cohérente avec les applications, afin d’obtenir une sauvegarde par snapshot rapide (en quelques secondes), une restauration rapide de la base de données (en quelques minutes), ainsi qu’une restauration, une récupération et un clonage à partir de snapshots ou de sauvegardes dans le coffre de stockage. Un snapshot est une copie à un instant donné des volumes de la base de données qui peut être créée en quelques secondes et ne consomme pas de stockage supplémentaire au moment de sa création. Il coexiste avec les volumes principaux de la base de données et peut être perdu si les volumes principaux sont compromis. Une sauvegarde vault est une copie du snapshot qui est stockée dans un stockage objet et dans un emplacement différent à des fins de reprise après sinistre.
* *Considérations relatives au RTO/RPO.* Lors de la mise en place de stratégies de sauvegarde de base de données, il est important de prendre en compte les exigences en matière d’objectifs de temps de récupération (RTO) et d’objectifs de point de récupération (RPO). Bien que l’impact sur les performances de la base de données soit minime avec une sauvegarde basée sur des instantanés, il existe un compromis entre la fréquence des sauvegardes, qui affecte le RTO/RPO, et les dépenses de stockage. Des sauvegardes plus fréquentes peuvent aider à atteindre des RTO/RPO plus faibles, mais peuvent augmenter les coûts de stockage. Il est important de trouver le bon équilibre en fonction des besoins de l’entreprise et du budget. La solution d’automatisation fournit un ensemble d’outils d’automatisation basé sur un playbook Ansible pour gérer la sauvegarde de base de données Oracle avec une rétention et une planification des sauvegardes configurables par l’utilisateur.

. Un compte Google Cloud a été configuré, et les paramètres VPC et réseau nécessaires ont été créés dans le projet spécifique de votre compte Google Cloud.

=== Déploiement automatisé d'Oracle avec un playbook Ansible



==== Fichiers de paramètres d'automatisation

[%collapsible]
====
Le playbook Ansible exécute les tâches d'installation et de configuration de la base de données avec des paramètres prédéfinis. Le kit d'outils prend actuellement en charge le déploiement de la base de données Oracle version 19c et 21c. Pour cette solution d'automatisation Oracle, trois fichiers de paramètres définis par l'utilisateur nécessitent une saisie de la part de l'utilisateur avant l'exécution du playbook.

* hôtes - définissez les cibles sur lesquelles le playbook d'automatisation s'exécute.
* vars/vars.yml - le fichier de variables globales qui définit les variables qui s'appliquent à toutes les cibles.
* host_vars/host_name.yml - le fichier de variables locales qui définit les variables qui s'appliquent uniquement à une cible nommée.  Dans notre cas d’utilisation, il s’agit des serveurs de base de données Oracle.


En plus de ces fichiers de variables définis par l'utilisateur, il existe plusieurs fichiers de variables par défaut qui contiennent des paramètres par défaut qui ne nécessitent aucune modification, sauf si nécessaire.  Les sections suivantes montrent comment configurer les fichiers de variables définis par l'utilisateur.

====


==== Configuration des fichiers de paramètres

[%collapsible]
====
. Cible Ansible `hosts` configuration du fichier :

###########################################
### ONTAP env specific config variables ###
###########################################

# Prerequisite to create three volumes in NetApp ONTAP storage from System Manager or cloud dashboard with following naming convention:
# {{ inventory_hostname}}_u01 or {{ inventory_hostname}}-u01 -- Oracle binary
# {{ inventory_hostname}}_u02 or {{ inventory_hostname}}-u02 -- Oracle data
# {{ inventory_hostname}}_u03 or {{ inventory_hostname}}-u03 -- Oracle redo
# It is important to strictly follow the name convention or the automation will fail.

host_datastores_nfs:
  - {vol_name: "{{ inventory_hostname}}-u01", lif: "{{ nfs_lif }}"}
  - {vol_name: "{{ inventory_hostname}}-u02", lif: "{{ nfs_lif }}"}
  - {vol_name: "{{ inventory_hostname}}-u03", lif: "{{ nfs_lif }}"}

###########################################
### Linux env specific config variables ###
###########################################

redhat_sub_username: "xxxxxxxxxx"
redhat_sub_password: "xxxxxxxxxx"


####################################################
### DB env specific install and config variables ###
####################################################

# Database version: support 19c and 21c, 19c|19.0.0 or 21c|21.0.0

==== Exécution du manuel de jeu

[%collapsible]
====
Il existe au total cinq playbooks dans la boîte à outils d’automatisation.  Chacun exécute des blocs de tâches différents et sert des objectifs différents.

....
0-all_playbook.yml - execute playbooks from 1-4 in one playbook run.
1-ansible_requirements.yml - set up Ansible controller with required libs and collections.
2-linux_config.yml - execute Linux kernel configuration on Oracle DB servers.
4-oracle_config.yml - install and configure Oracle on DB servers and create a container database.
5-destroy.yml - optional to undo the environment to dismantle all.
....
Il existe trois options pour exécuter les playbooks avec les commandes suivantes.

. Exécutez tous les playbooks de déploiement en une seule exécution combinée.
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml
----
. Exécutez les playbooks un par un avec la séquence de nombres de 1 à 4.
+
[source, cli]]
----
ansible-playbook -i hosts 1-ansible_requirements.yml -u admin -e @vars/vars.yml
----
+
[source, cli]
----
ansible-playbook -i hosts 2-linux_config.yml -u admin -e @vars/vars.yml
----
+
[source, cli]
----
ansible-playbook -i hosts 4-oracle_config.yml -u admin -e @vars/vars.yml
----
. Exécutez 0-all_playbook.yml avec une balise.
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t ansible_requirements
----
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t linux_config
----
+
[source, cli]
----
ansible-playbook -i hosts 0-all_playbook.yml -u admin -e @vars/vars.yml -t oracle_config
----
. Défaire l'environnement
+
[source, cli]
----
ansible-playbook -i hosts 5-destroy.yml -u admin -e @vars/vars.yml
----

=== Sauvegarde de base de données Oracle avec Google Cloud NetApp Volumes



==== Instantané de la base de données Oracle et sauvegarde de coffre-fort

[%collapsible]
====
Pour faciliter la configuration facile de la sauvegarde de la base de données Oracle, l'équipe Solutions Engineering de NetApp a développé un playbook Ansible pour automatiser la sauvegarde de la base de données Oracle avec une rétention et une planification des sauvegardes configurables par l'utilisateur. Le playbook exploite les fonctionnalités de snapshot et de sauvegarde de coffre-fort de Google Cloud NetApp Volumes pour réaliser une sauvegarde par snapshot rapide (en quelques secondes), une restauration, une récupération et un clonage rapides (en quelques minutes) de la base de données à partir de snapshots ou de sauvegardes dans le coffre de stockage.

. Clonez une copie du kit d'outils d'automatisation de sauvegarde de base de données Oracle NetApp pour GCNV.
+
[source, cli]
----
git clone https://bitbucket.ngage.netapp.com/scm/ns-bb/na_oracle_bkup_gcnv.git
----
+

NOTE: La boîte à outils n'est accessible qu'aux utilisateurs internes de NetApp disposant d'un accès Bitbucket pour le moment.  Pour les utilisateurs externes intéressés, veuillez demander l'accès à votre équipe de compte ou contacter l'équipe d'ingénierie des solutions NetApp .

. Lisez le fichier README du kit d'outils et suivez les instructions ci-dessous pour configurer et planifier la tâche de sauvegarde à partir du crontab ou de tout autre outil de planification. Le playbook est conçu pour être exécuté sur un nœud de contrôleur Ansible ayant accès aux machines virtuelles du serveur de base de données Oracle et à Google NetApp Volumes. Il créera des instantanés cohérents au niveau de l'application des volumes de données de la base de données selon la planification et la politique de rétention définies, puis copiera les instantanés dans le coffre pour des besoins de reprise après sinistre.
. Par défaut, le playbook crée une sauvegarde instantanée quotidienne et des instantanés horaires chaque heure. La rétention par défaut des instantanés est de 7 instantanés quotidiens et 24 instantanés horaires. Tout instantané supplémentaire dépassant la rétention est supprimé et une copie continue de 7 instantanés quotidiens et 24 instantanés horaires est maintenue. Vous pouvez ajuster la fréquence et la rétention des sauvegardes en fonction de vos exigences RTO/RPO et des considérations de coût de stockage. Alors que la sauvegarde instantanée quotidienne couvre tous les volumes de données de la base de données, la sauvegarde instantanée horaire ne concerne que les volumes de journaux et permet d'économiser de l'espace de stockage. Lors de la sauvegarde instantanée quotidienne, le playbook supprime également les fichiers journaux d'archive Oracle en fonction de la rétention définie afin d'économiser de l'espace de stockage sur les volumes de journaux de la base de données.
. Voici un exemple d'entrée crontab pour créer une sauvegarde instantanée et la copier dans le coffre-fort.
+
....

[admin@ansiblectl na_oracle_bkup_gcnv]$ crontab -l
0 0 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_snapshot_daily.sh
0 1-23 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_snapshot_hourly.sh
5 0 * * 7 /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_weekly.sh
5 0 * * 1-6 /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_daily.sh
5 1-23 * * * /home/admin/na_oracle_bkup_gcnv/oracle_standalone_vaultbkup_hourly.sh
....
. La sauvegarde du coffre-fort peut être exécutée à un intervalle hebdomadaire, quotidien ou horaire selon vos exigences RTO/RPO. Alors que les sauvegardes hebdomadaires et quotidiennes incluent tous les volumes de base de données, la sauvegarde horaire du coffre-fort n'inclut que les volumes de journaux afin d'économiser de l'espace de stockage. La toute première sauvegarde du coffre-fort prend plus de temps car elle crée la base de référence. Une fois la sauvegarde de référence établie, toutes les sauvegardes supplémentaires du coffre-fort sont incrémentielles en utilisant la méthodologie incremental forever. Toutes les sauvegardes du coffre-fort sont créées à partir des instantanés cohérents avec l'application les plus récents au moment de l'exécution afin d'assurer la possibilité de restauration. Contrairement aux sauvegardes de référence et incrémentielles classiques, les données de la sauvegarde de référence du coffre-fort sont intégrées à chaque sauvegarde incrémentielle. En d'autres termes, chaque sauvegarde incrémentielle du coffre-fort contient l'ensemble des données et peut être utilisée pour la restauration sans qu'il soit nécessaire de restaurer la sauvegarde de référence. Cette approche simplifie la gestion des sauvegardes et le processus de restauration tout en assurant une utilisation efficace du stockage dans le coffre-fort. Il n'est pas nécessaire de se soucier de la chaîne de sauvegardes et des dépendances lorsque vous devez supprimer une sauvegarde, car toutes les sauvegardes sont indépendantes avec cette approche. Le script d'automatisation des sauvegardes supprime automatiquement les sauvegardes pour respecter l'objectif de rétention défini.

=== Restauration et clonage de la base de données Oracle avec Google Cloud NetApp Volumes

[%collapsible]
====
La restauration à un point précis dans le temps d'une base de données Oracle est couramment utilisée pour récupérer des données supprimées ou corrompues accidentellement, ou pour corriger une erreur logique. Avec les snapshots Google NetApp Volumes, vous pouvez facilement effectuer une restauration à un point précis dans le temps d'une base de données Oracle en restaurant la base de données à un snapshot spécifique. Cela vous permet de récupérer rapidement d'une perte de données ou d'une corruption sans avoir à restaurer à partir d'une sauvegarde complète. La procédure suivante illustre les étapes pour récupérer une table supprimée à l'aide des snapshots Google NetApp Volumes.

==== Restauration de la base de données Oracle sur un nouvel hôte à l'aide des sauvegardes de coffre Google Cloud NetApp Volumes

[%collapsible]
====
En cas de panne nécessitant une restauration sur un nouvel hôte, comme lorsque l'hôte d'origine n'est plus disponible et que les volumes principaux de la base de données sont inaccessibles, vous pouvez utiliser les sauvegardes vault de Google Cloud NetApp Volumes pour restaurer la base de données Oracle sur le nouvel hôte. Le processus est similaire à la restauration sur place à l'aide d'instantanés, mais au lieu de revenir à un instantané, vous restaurerez la base de données à partir d'une sauvegarde vault. Cela vous permet de restaurer la base de données sur un hôte différent, ce qui peut être utile dans les scénarios où l'hôte d'origine n'est plus disponible ou a subi une défaillance matérielle. Les étapes pour restaurer à partir d'une sauvegarde vault sont les suivantes :

. Identifiez la sauvegarde du coffre-fort qui contient les données que vous souhaitez récupérer. Vous pouvez utiliser la Google Cloud Console ou l’outil en ligne de commande gcloud pour lister les sauvegardes du coffre-fort disponibles pour votre volume de base de données Oracle. Cliquez sur les trois points à la fin de la liste des sauvegardes du coffre-fort et sous `Show more` pour afficher les options. Sélectionnez le `Create new volume from backup` pour restaurer à partir de la sauvegarde du coffre-fort sélectionnée. Répétez pour tous les volumes de base de données. Vous avez également la possibilité de restaurer dans le même pool de stockage ou dans un pool de stockage différent si nécessaire.

La base de données restaurée peut avoir des fichiers de contrôle incohérents après la restauration de la base de données. Dans ce cas, vous pouvez utiliser le fichier de contrôle situé dans le volume de journal pour écraser le fichier de contrôle dans le volume de données afin de résoudre le problème.
